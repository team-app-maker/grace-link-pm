# GraceLink IA / Menu Structure

> SSOT 연계 문서: `docs/SSOT.md`

## 1. 문서 목적

이 문서는 GraceLink의 현재 정보구조(IA), 메뉴 구조, 라우트 구조를 정의하는 **공식 구조 문서**다.

이 문서의 목적은 세 가지다.
- 사용자가 앱 안에서 어떻게 이동하는지 기준선을 만든다.
- 현재 노출 구조와 숨김 구조를 명확히 분리한다.
- 이후 홈 개편, 탭 개편, 메뉴 개편 시 기준 문서로 사용한다.

## 2. 현재 IA 한눈에 보기

### 2.1 승인 전
```text
Splash
  → Login
    → Onboarding STEP 1 PASS
    → STEP 2 기본 정보
    → STEP 3 교회 등록
    → STEP 4 태그
    → STEP 5 말씀/찬양
    → STEP 6 신앙 질문
    → STEP 7 제출 전 확인
      → Under Review
```

### 2.2 승인 후
```text
탭 4개
- 홈
- 오늘의 소개
- 채팅
- 내 정보

숨김 핵심 라우트
- 인박스
- 보관함
- 스토어
- 상대 프로필 상세
- 내 정보 수정
- 채팅 상세
- 운영 심사
```

## 3. 현재 메뉴 구조

## 3.1 하단 탭 (실제 노출)

| 탭명 | 현재 라우트 | 역할 | 현재 평가 |
| --- | --- | --- | --- |
| 홈 | `/(tabs)` | 말씀/공지 허브 | 브랜드 톤은 좋지만 행동성이 약함 |
| 오늘의 소개 | `/(tabs)/reco` | 추천 카드 소비 | 현재 핵심 탐색 화면 |
| 채팅 | `/(tabs)/chat` | 열린 채팅방 관리 | 상태 추적 역할 명확 |
| 내 정보 | `/(tabs)/settings` | 프로필/필터/상점/운영 메뉴 | 너무 많은 기능이 몰려 있음 |

## 3.2 숨김이지만 핵심인 메뉴/화면

| 화면 | 현재 진입 방식 | 문제 |
| --- | --- | --- |
| 인박스 | 헤더/내부 링크 | 탭급 중요도인데 발견성이 낮음 |
| 보관함 | 내 정보 내부 메뉴 | 추천 경험과 가까운데 너무 깊다 |
| 스토어 | `/shop` 또는 숨김 탭 | 결제/광고가 핵심인데 직접 진입성이 약함 |
| 운영 심사 | 내 정보 메뉴 | 사용자용 메뉴 구조 안에 섞여 있음 |

## 4. 현재 라우트 구조

## 4.1 루트 레벨

| 라우트 | 역할 |
| --- | --- |
| `/` | 인증/상태 분기 |
| `/(auth)` | 로그인/온보딩/심사대기 |
| `/(tabs)` | 승인 후 메인 |
| `/profile/[targetUuid]` | 상대 프로필 상세 |
| `/profile/edit` | 내 정보 수정 |
| `/chat/[chatId]` | 채팅 상세 |
| `/admin/review` | 운영 심사 |
| `/shop` | 독립 스토어 |

## 4.2 승인 전 라우트

| 단계 | 라우트 | 역할 |
| --- | --- | --- |
| Splash | `/(auth)/splash` | 첫 진입 브랜딩 |
| Login | `/(auth)/login` | SSO 로그인 |
| STEP 1 | `/(auth)/ob-01` | PASS 인증 |
| STEP 2 | `/(auth)/ob-02` | 기본 정보 |
| STEP 3 | `/(auth)/ob-03` | 교회 등록 |
| STEP 4 | `/(auth)/ob-05` | 태그 |
| STEP 5 | `/(auth)/ob-06` | 말씀/찬양 |
| STEP 6 | `/(auth)/ob-07` | 신앙 질문 |
| STEP 7 | `/(auth)/ob-08` | 제출 전 확인 |
| Review | `/(auth)/under-review` | 심사 대기 |

## 4.3 승인 후 라우트

| 클러스터 | 라우트 | 역할 |
| --- | --- | --- |
| 홈 | `/(tabs)` | 상태/콘텐츠 허브 |
| 추천 | `/(tabs)/reco` | 추천 탐색 |
| 인박스 | `/(tabs)/inbox` | 응답 대기/발신 현황 |
| 채팅 목록 | `/(tabs)/chat` | 대화방 목록 |
| 채팅 상세 | `/chat/[chatId]` | 메시지 송수신 |
| 내 정보 | `/(tabs)/settings` | 프로필/필터/기타 메뉴 |
| 보관함 | `/(tabs)/vault` | 카드 재진입 |
| 스토어 | `/shop`, `/(tabs)/shop` | 만나 충전/광고 |
| 프로필 상세 | `/profile/[targetUuid]` | 매칭 진입 |
| 프로필 수정 | `/profile/edit` | 자기 정보 수정 |
| 운영 심사 | `/admin/review` | risk review |

## 5. 현재 구조의 문제

### 5.1 홈이 허브 역할을 충분히 못 한다
- 말씀/공지 자체는 브랜드 톤과 맞는다.
- 하지만 핵심 행동으로 이어지는 우선순위 정리는 약하다.

### 5.2 핵심 기능이 너무 숨겨져 있다
- 인박스, 보관함, 스토어는 제품 핵심이다.
- 그런데 현재 탭이나 1차 메뉴에서 직접 드러나지 않는다.

### 5.3 내 정보 화면에 과도한 기능이 몰려 있다
- 프로필
- 만나
- 필터
- 보관함
- 운영 심사
- 로그아웃

사용자 관점과 운영 관점이 같은 메뉴 레벨에 섞여 있다.

### 5.4 스토어 진입 경로가 중복된다
- `/shop`
- `/(tabs)/shop`

기능은 거의 같고, 진입 구조만 다르다.

## 6. 권장 구조 방향

## 6.1 최종 방향

### 원칙
- 홈은 **행동 허브**
- 오늘의 소개는 **탐색 화면**
- 인박스는 **응답 허브**
- 채팅은 **관계 유지 화면**
- 내 정보는 **설정/프로필 화면**

### 추천 탭 구조 (차기안)
| 탭 | 역할 |
| --- | --- |
| 홈 | 오늘 해야 할 일 / 상태 허브 |
| 소개 | 오늘의 소개 / 카드 탐색 |
| 인박스 | 받은 신청 / 보낸 신청 |
| 채팅 | 대화 |
| 내 정보 | 프로필 / 설정 |

### 보관함 / 스토어 처리 권장
- 보관함: 홈 또는 내 정보에서 보조 진입
- 스토어: 내 정보 아래보다, paywall/잔액 영역과 직접 연결

## 6.2 최소 개편안

탭 구조를 당장 안 바꾼다면:
- 홈을 행동 허브로 개편
- 홈에서 인박스/오늘의 소개/보관함/스토어 상태를 한 번에 보여주기
- 내 정보에서는 운영 심사 진입을 일반 사용자 메뉴와 시각적으로 분리하기

## 7. 권장 IA 카피

| 현재 | 권장 |
| --- | --- |
| 오늘 | 홈 |
| 오늘의 소개 | 소개 |
| 내 정보 | 내 정보 또는 프로필 |
| 운영 심사 콘솔 | 운영 심사 |

### 홈 헤더 추천
- `오늘의 흐름`
- `지금 해야 할 일`
- `오늘의 진행 상황`

가장 추천하는 것은 **홈 탭 + 오늘의 흐름 헤더** 조합이다.

## 8. 메뉴 개편 시 체크리스트

- 탭 수가 늘면 현재 사용자 행동이 더 빨라지는가?
- 인박스/보관함/스토어 발견성이 높아지는가?
- 홈이 브랜드 화면이 아니라 제품 허브가 되는가?
- 운영 메뉴가 사용자 메뉴와 구분되는가?
- `/shop` 과 `/(tabs)/shop` 중복이 해소되는가?

## 9. 결론

현재 구조는 기능 자체는 갖췄지만, 사용자가 핵심 경험을 자연스럽게 따라가도록 정돈되어 있지는 않다.

GraceLink의 다음 IA 목표는 단순히 탭을 바꾸는 것이 아니라,
**브랜드 톤을 유지하면서도 행동 우선순위가 보이는 구조**를 만드는 것이다.
